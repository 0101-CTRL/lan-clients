# APIv3 LAN Clients Console

A standalone local web UI for exploring the internal-beta NCM APIv3 `lan_clients` endpoint.

```text
GET https://api.cradlepointecm.com/api/v3/beta/lan_clients
```

The app runs locally on your server. Your browser talks to this local FastAPI app, and the FastAPI app calls NCM using the bearer token you paste into the UI.

## What this tool is for

This utility helps you inspect LAN client records without rebuilding curl commands by hand.

It can:

- Check whether your bearer token can reach the `lan_clients` endpoint.
- Fetch LAN clients from `/api/v3/beta/lan_clients`.
- Filter by router ID, hostname, and network type.
- Separate recently seen records from older / previously seen records.
- Show timestamp context for `collection_time` versus `updated_at`.
- Explain ambiguous endpoint fields with hover help.
- Show the actual GET query history generated by the UI.
- Export shown rows or fetched rows as CSV.
- Export the raw API response as JSON.
- Generate a copyable, redacted curl command.

## Important endpoint notes

The `lan_clients` endpoint is internal beta and is not available to all customers. A `403 Forbidden` usually means the token/account does not have access to this beta endpoint.

The endpoint documentation lists `GET 201 Created` as a success response. This console treats both `200` and `201` as success.

This endpoint should be treated as a **known / last-seen LAN client inventory**, not a guaranteed live connected-client list. The endpoint does not provide a clean `connected: true/false` field. The UI therefore uses timestamps such as `collection_time` and, when needed, `updated_at` to make a lower-confidence activity inference.

For wireless clients, `rxrate` and `txrate` are last-reported values based on the data-update interval. They are not live throughput readings.

## ELI5 install: manual run

Use this when you just want to run the UI in a terminal.

### 1. Install basic packages

```bash
sudo apt update
sudo apt install -y git python3 python3-venv python3-pip
```

### 2. Clone the repo

```bash
cd /home/ubuntu
git clone https://github.com/0101-CTRL/lan-clients.git
cd lan-clients
```

### 3. Start the app

```bash
PORT=8093 ./run_local.sh
```

### 4. Open the UI

From your browser, go to:

```text
http://<server-ip>:8093
```

For example:

```text
http://192.168.1.50:8093
```

Leave the terminal open while you use the app. To stop it, press:

```text
Ctrl+C
```

## ELI5 install: run it as a systemd service

Use this when you want the app to keep running in the background and automatically restart after a reboot.

### 1. Clone the repo

```bash
cd /home/ubuntu
git clone https://github.com/0101-CTRL/lan-clients.git
cd lan-clients
```

### 2. Move it to `/opt`

```bash
sudo rm -rf /opt/api-v3-lan-clients-ui
sudo mkdir -p /opt/api-v3-lan-clients-ui
sudo cp -a . /opt/api-v3-lan-clients-ui/
sudo chown -R ubuntu:ubuntu /opt/api-v3-lan-clients-ui
```

If your Linux username is not `ubuntu`, replace `ubuntu:ubuntu` with your own username and group.

You can check your username with:

```bash
whoami
```

You can check your group with:

```bash
id -gn
```

### 3. Install the service

The easiest method is the included installer:

```bash
cd /opt/api-v3-lan-clients-ui
sudo ./install_systemd.sh
```

By default this creates a service named:

```text
api-v3-lan-clients-ui
```

It runs on port:

```text
8093
```

### 4. Check that it started

```bash
sudo systemctl status api-v3-lan-clients-ui --no-pager -l
```

You want to see something like:

```text
Active: active (running)
```

### 5. Open the UI

```text
http://<server-ip>:8093
```

## Manual systemd install

If you do not want to use `install_systemd.sh`, you can install the example service manually.

```bash
cd /opt/api-v3-lan-clients-ui
sudo cp api-v3-lan-clients-ui.service.example /etc/systemd/system/api-v3-lan-clients-ui.service
sudo systemctl daemon-reload
sudo systemctl enable --now api-v3-lan-clients-ui
sudo systemctl status api-v3-lan-clients-ui --no-pager -l
```

Important: check the `User=` and `Group=` lines in the service file. They need to match a real Linux user that can read and write `/opt/api-v3-lan-clients-ui`.

## Data, tokens, and history

This app does not intentionally write bearer tokens, GET history, or fetched client data to server-side database files.

The optional remembered token and GET history are browser-local. To clear them, use the UI clear buttons or clear site data in the browser.

In Chrome DevTools, while on the app page, this also clears browser-local state:

```javascript
localStorage.clear();
sessionStorage.clear();
```

## Troubleshooting

### The page does not load

Check whether the app is running:

```bash
sudo systemctl status api-v3-lan-clients-ui --no-pager -l
```

If you are running manually, make sure the terminal still shows the app running.

### The service fails immediately

Check logs:

```bash
sudo journalctl -u api-v3-lan-clients-ui --no-pager -n 80
```

The most common causes are:

- The service `User=` does not exist.
- The service user cannot write to the app directory.
- Python virtual environment creation failed.
- Outbound HTTPS to `api.cradlepointecm.com` is blocked.

### 403 Forbidden

The endpoint is internal beta and may not be enabled for the account/token.

### 400 Bad Request

Check filter values. The documented network type values are:

```text
ethernet, wifi, bluetooth, unknown
```

### No clients returned

Remove filters and fetch again. The endpoint represents clients last known to the router, so a narrow router, hostname, or network type filter can make the result look empty.

### API fetch works, but the timestamp does not change

That usually means the app fetched the latest stored data from NCM, but NCM has not received a newer LAN-client report for that router yet. The UI separates **Last API fetch** from the endpoint’s client record timestamp for this reason.
